Methodology & data sources

This page documents how leads in the dashboard are produced. Every tag a lead carries is traceable to a public-records source, a join rule, and a date. No skip trace, no AVM, no MLS, no proprietary scoring.

Tag taxonomy

Each lead carries a tags[] array. Tags fall into three categories — distress, owner profile, and derived. Tier is decided by the distress tag count alone.

Tier logic

Tier is decided by the distress tag count only. Owner-profile and derived tags are filters, not tier promoters.

  • Hot — 3 or more distinct distress tags
  • Warm — 2 distinct distress tags
  • Active — 1 distinct distress tag
  • Parcels with zero distress tags are dropped from records[] entirely.

A score sum can never promote tier. Sub-flags affect in-tier sort order only.

Diff logic

On every refresh, the pipeline diffs the current run against the previous run's snapshot at data/leads.previous.json. Each record carries a _diff_status field:

  • new — PID not present in the previous file
  • newly_tagged — PID present in both files but with a different tags[] array
  • existing — PID present in both files with the same tags

The Today view shows only new + newly_tagged rows; the All Leads view shows the full database.

Baseline reset behavior

When the previous file is absent or carries the pre-tag-taxonomy schema, the pipeline:

  1. Archives the old previous file under data/raw/archive/leads.previous.<timestamp>.json
  2. Sets every record to _diff_status = "existing"
  3. Sets new_count = newly_tagged_count = 0
  4. Logs "Diff baseline established."
  5. Snapshots today's data to leads.previous.json so tomorrow's run does a real diff

The Today view's empty state on a baseline run reads: "Diff baseline established. New leads appear here starting tomorrow."

Join logic

Join quality is the highest priority. Bad joins are worse than missing tags.

Join priority, top-down:

  1. Exact PID match (PARID format R#####-###-###-###)
  2. Normalized parcel ID match (PIN, MAPIDKEY)
  3. Exact normalized property-address match — only when PID is missing on the source
  4. Fuzzy address match — only when street number, normalized street name, city, and state all match
  5. Owner name alone is never sufficient
  6. Grantor / grantee / decedent name alone is never sufficient
  7. Case number alone is never sufficient unless the case payload also includes matching PID or property address
  8. If multiple parcels could match one signal, do not attach unless one match is clearly stronger
  9. If confidence is unclear, do not emit the tag

Address normalization, applied before every comparison:

  • Uppercase, trim, remove punctuation, collapse whitespace
  • Standardize street suffixes (STREET → ST, AVENUE → AVE, ROAD → RD, etc.)
  • Compare address line + city + state when available

Document codes are normalized before matching: uppercase, strip non-alphanumerics, collapse whitespace. So M/L matches ML, LIS PENDENS matches LISPENDENS, etc.

Two-Truths invariant

The pipeline calls two_truths_check() immediately before writing leads.json. It recomputes tier_counts and tag_counts from records[] and compares to the header values. Any mismatch raises and the file is not written. This guards against the most common dashboard failure mode: counts and visible records computed from different code paths drifting.

The dashboard reuses the same tags[] field for both pill counts and the rendered table — there is one matches(lead) function in index.html and it powers the stat row, every pill count, the table body, and the CSV export.

Data limitations (honest disclosure)

Most ROD-derived tags currently fire on zero leads in this build. NHC Register of Deeds records carry no parcel ID, and the description field carries only subdivision + lot + block — not a street address. Per the join safety rules, name-only joins (grantor/grantee/decedent → owner) are insufficient. Until either the source data exposes a PID or richer address context, ROD signals are surfaced only in the raw JSONL files for manual review, not auto-attached to parcels.
  • No public code-enforcement dataset — NHC and the City of Wilmington do not publish code-enforcement cases. Demolition Order is the only "code" signal, sourced from EnerGov building permits with type containing "Demolition".
  • No vacancy registry feed — no public NHC or Wilmington vacancy registry. Vacancy is not a tag in this build.
  • No City of Wilmington EnerGov — Wilmington's separate EnerGov SelfService instance is gated behind a Tyler SPA + CSRF token. City-issued building permits and city code cases are not in this build.
  • eCourts (Tyler Odyssey) is gated — AWS WAF CAPTCHA every 5 minutes for both anonymous and registered users. Foreclosure SP filings and estate E openings come from the StarNews newspaper-notice flow as the parallel feed (NC statute requires statutory publication, so the events surface there).
  • "Free & Clear" suppressed — requires a recent-DoT-absent inference, but rod_deed_of_trust.jsonl is not pulled by any scraper in this repo. Tag globally suppressed; not rendered.
  • "Senior Owner" suppressed — NHC PropertyOwners has no senior/elderly exemption field with meaningful coverage (≤ 5%). Tag globally suppressed; not rendered.
  • Beach towns (Wrightsville Beach, Carolina Beach, Kure Beach) — no public open-data portals; town-issued permits not in any open feed.
  • "Hot" tier may be empty on any given day. With current accessible data sources, no parcel currently stacks 3 or more distinct distress tags. This is a real-data outcome, not a pipeline bug — the tag audit at data/raw/tag_audit.json documents which sources contributed how many leads to each tag.

What's not included (and why)

  • No skip trace — phone / email columns in the CSV export are placeholders for downstream skip-trace vendors.
  • No AVM / MLS / mortgage balance / lender data — out of scope; assessed value (APRTOT) is the only value baseline.
  • No year-built or building age — NHC PropertyOwners feed doesn't expose it.
  • No ML scoring — every tag is rule-driven and explainable. Tier is a count of distinct distress tags, nothing more.

Daily refresh

The pipeline is built for daily live data. pipeline/refresh.py runs every scraper in dependency order, regenerates leads.json, runs the Two-Truths check, snapshots the result to leads.previous.json for tomorrow's diff, and pushes to GitHub. Windows Task Scheduler (config in scripts/daily_refresh.xml) runs it at 04:00 America/Chicago every day. HEARTBEAT.json in the repo root tracks per-source last_success for operations alerting.
